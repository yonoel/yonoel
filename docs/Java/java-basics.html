Java Basic

最早是在培训班学习java开发，自己自学黑马的10天javase的课程。 这个课程挺好的，花了5天啃完了，一共刷了2遍，尤其是lambda，方法引用，oop这一块。

内容包括

  • javase（standard edition）
    • 数据类型，拆装箱，控制语句，包，类，接口，对象，方法，注解，反射
  • javaee（enterprise edition）
    • Servlet,JSP,JDBC,tomcat,mysql
  • SSM，SSH
    • SpringMVC,Hibernate,MyBatis,Spring,Structs

java se#

这一块都是基础，牢记，经常使用。

oop的三大特征

  1. 多态
  2. 封装
  3. 继承

多态和继承更多的是描述对象之间的关系，而封装归纳的是对象与对象，甚至对象与流程之间的关系。

最早的编程思想是面向过程，也就是把一切行为自动化，后来大佬们觉得，好累，一天到晚写个脚本，有没有点出息！我们是世界上的聪明人，不想这么辛苦！达咩！然后就提出了新的编程思想，面向对象，（也可能是因为太忙了，没有对象。。。）。一言以蔽之，就是万物皆为对象。十分理想化的思想。

java ee#

这块我没有太多开发经历，源码也看的不多，自己稍微归纳一下重点。

jsp就是页面，用java代码写页面，或者用jsp的表达式。

servlet是个容器，其次是个web容器，也就是后端，用来处理http请求，链接了web客户端。

tomcat是个服务器端，是用来管理servlet容器的。

当时的开发，就是写一个类继承httpservlet，然后打包成war包，放在tomcat里，tomcat会解压缩该war包，tomcat作为根路径，就会转发请求到该war包里的程序。war的配置目录便是WEB-INF目录，其中web.xml很关键，控制了该war包下的路由，过滤器，拦截器，web层的配置基本在这里。

事实证明，18年javaee已经out了，市场上正在大力推动前后端分离，单体应用正在逐渐消亡。

什么是前后端分离？#

大家一致认同的前后端分离的例子就是SPA(Single-page application)，所有用到的展现数据都是后端通过异步接口(AJAX/JSONP)的方式提供的，前端只管展现。

为什么要前后端分离?#

  1. 现有开发模式的适用场景

比如后端为主的MVC，做一些同步展现的业务效率很高，但是遇到同步异步结合的页面，与后端开发沟通起来就会比较麻烦。Ajax为主SPA型开发模式，比较适合开发APP类型的场景，但是只适合做APP，因为SEO等Search Engine Optimization问题不好解决，对于很多类型的系统，这种开发方式也过重。

  1. 前后端职责不清

在业务逻辑复杂的系统里，我们最怕维护前后端混杂在一起的代码，因为没有约束，M-V-C每一层都可能出现别的层的代码，日积月累，完全没有维护性可言。 虽然前后端分离没办法完全解决这种问题，但是可以大大缓解。因为从物理层次上保证了你不可能这么做。

  1. 开发效率问题

淘宝的Web基本上都是基于MVC框架webx，架构决定了前端只能依赖后端, 依赖后端数据，效率依然不高。 另外，后端也没法摆脱对展现的强关注，从而专心于业务逻辑层的开发。

  1. 对前端发挥的局限

性能优化如果只在前端做空间非常有限，于是我们经常需要后端合作才能碰撞出火花，但由于后端框架限制，我们很难使用Comet、Bigpipe等技术方案来优化性能。为了解决以上提到的一些问题，我们进行了很多尝试，开发了各种工具，但始终没有太多起色，主要是因为我们只能在后端给我们划分的那一小块空间去发挥。只有真正做到前后端分离，我们才能彻底解决以上问题。

怎么做前后端分离？#

SPA确实做到了前后端分离，但这种方式存在两个问题：

  • WEB服务中，SPA类占的比例很少。很多场景下还有同步/同步+异步混合的模式，SPA不能作为一种通用的解决方案。
  • 现阶段的SPA开发模式，接口通常是按照展现逻辑来提供的，有时候为了提高效率，后端会帮我们处理一些展现逻辑。

这就意味着后端还是涉足了View层的工作，不是真正的前后端分离。 SPA式的前后端分离，是从物理层做区分（认为只要是客户端的就是前端，服务器端的就是后端）

  • 前端：负责View和Controller层。
  • 后端：负责Model层，业务处理/数据等。
